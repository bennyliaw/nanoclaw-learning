# Plan: Use Case 1.2 Phase A — Google One Storage Audit (Nano-orchestrated)
_v1 — initial plan, pre-revision_

## Context

Use case 1.2 Phase A: a chat-triggered audit that scans Google Photos and Google Drive,
identifies cleanup candidates, and reports via Telegram. No deletions — read-only audit.

**Architecture decision:** Nano (dm-with-bigben) orchestrates this. `audit.py` lives in
Nano's group folder. No new agent group or wiring needed. When you later want a separate
Telegram chat or WhatsApp routing, the migration is: create `google-one-cleanup` group,
move `audit.py` there, add a wiring — nothing in the script changes.

**What this teaches:**
- OneCLI credential proxy for Google OAuth (no auth code needed in the script)
- Python3 in a NanoClaw container (requires apt packages + container rebuild)
- Agent-driven multi-step operation: run script → process output → send report
- Audit-before-act pattern

---

## Step 0: User Action — Verify Prerequisites

**Must happen before writing or running any code.**

### 0a. Confirm OneCLI supports Google Photos + Drive

Open the OneCLI web UI and check the Apps list:
```
http://127.0.0.1:10254  →  Apps
```
Look for Google Photos and Google Drive as available providers. If they aren't listed,
the plan needs to be revised (different auth path or Drive-only audit).

### 0b. Connect both services
Connect:
- **Google Photos** (readonly scope)
- **Google Drive** (readonly scope — needed for storage quota totals)

The OneCLI gateway auto-injects `HTTPS_PROXY` into every container, so once services are
connected in the web UI, the audit script gets authenticated requests transparently — no
API keys, no token handling in the script itself.

### 0c. Probe the Photos API before building

⚠️ **Known risk:** Google restricted the Photos Library API around March 2025.
`mediaItems.list` may now return only **app-created** content, not your full library.
If that's the case, the planned inventory approach doesn't work.

**Probe first — before writing audit.py:**
```bash
# From inside the container or via the agent running a one-liner:
curl "https://photoslibrary.googleapis.com/v1/mediaItems?pageSize=10"
```
- If it returns real photos from your library → proceed as planned
- If it returns empty or only app-created items → revisit: Picker API, Google Takeout export, or Drive-only audit

## Step 1: Add Python3 to Nano's Container

Nano's container is Node 22 — Python3 must be added explicitly.

```bash
ncl groups config add-package --id ag-1780312519263-8b8bto --apt python3
ncl groups config add-package --id ag-1780312519263-8b8bto --apt python3-pip
ncl groups restart --id ag-1780312519263-8b8bto --rebuild
```

`requests` is installed by the script itself on first run (lightweight bootstrap pattern,
same approach as `typhoon_monitor.py`). No need to add it to the package list.

## Step 2: Write `groups/dm-with-bigben/audit.py`

Script responsibilities:
1. **Storage totals** — Drive `about` endpoint:
   `GET https://www.googleapis.com/drive/v3/about?fields=storageQuota`
   Returns: `limit`, `usage`, `usageInDrive`, `usageInGmail`. Photos usage is rolled into
   `usage` (no separate field since 2021 Google storage policy change).

2. **Media inventory** — Photos `mediaItems` with full pagination:
   `GET https://photoslibrary.googleapis.com/v1/mediaItems?pageSize=100`
   Paginate until no `nextPageToken`. Collect: `id`, `filename`, `mimeType`,
   `mediaMetadata.creationTime`, `mediaMetadata.photo`/`video`.

   ⚠️ Contingent on Step 0c probe confirming the API returns your full library.

3. **Analysis (metadata-only, v1):**
   - **Burst sequences:** items within 2 seconds of each other, same mimeType
   - **Screenshots:** filename matches `Screenshot_*`, `screen*`, `*Screenshot*`
   - **Large videos:** `mediaMetadata.video` present (flag all — duration not in v1 API)
   - **Old content:** items older than 3 years
   - **WhatsApp media:** filename matches `IMG-*-WA*` or `VID-*-WA*`

4. **Output:** Write `/workspace/agent/audit_results.json`. Print a compact JSON summary
   to stdout for the agent to read (counts + top 5 examples per category).

5. **Auth error handling:** If any API call returns 401/407, print the error JSON (OneCLI
   may include a `connect_url` field in auth errors) and exit with code 1. The agent reads
   this and shows the user the link to connect the service.
   Note: exact error shape TBC against first real auth failure — adapt if needed.

**Bootstrap pattern** (top of script, same as typhoon_monitor.py):
```python
for pkg in ["requests"]:
    try:
        __import__(pkg)
    except ImportError:
        subprocess.check_call([sys.executable, "-m", "pip", "install", "-q", pkg], ...)
```

## Step 3: Update `groups/dm-with-bigben/CLAUDE.local.md`

Add an **Audit** section to the existing CLAUDE.local.md (do not replace it). The section
should specify:
- **Trigger phrases:** "run audit", "storage audit", "check storage", "audit"
- **On trigger:** Run `python3 /workspace/agent/audit.py`
- **On exit code 1:** Show the error/connect_url from stdout, ask user to authorize in browser, then retry
- **On success:** Read `/workspace/agent/audit_results.json`, format Telegram report:
  - Header: used / quota / percentage (e.g. "182 GB / 200 GB — 91%")
  - Section per category: count + estimated impact + top 3–5 filenames
  - Footer: "Reply 'start cleanup' when ready for Phase B"
- **Never** modify, delete, or move any Google files

---

## Step 4: First Run

1. Confirm NanoClaw is running
2. Send "run audit" in your Telegram chat to Nano
3. If Google not connected → agent shows connect_url → authorize → retry
4. Audit runs (~1–3 min for large libraries) → report in Telegram

## Verification

1. **OneCLI provider check (Step 0a):** Confirm Google Photos + Drive appear in Apps list
2. **API probe (Step 0c):** One `mediaItems?pageSize=10` call confirms full library is accessible
3. **Auth path:** Send "run audit" before connecting Google → confirm agent shows
   connect_url (not a raw error, not asking for credentials)
4. **Happy path:** After connecting, audit completes → report in Telegram with storage
   numbers plausibly matching your ~90% full Google One account
5. **Read-only check:** Nothing in the script or agent instructions can modify Google data

## Future Migration Path (when ready for separate routing)

When you want the cleanup agent on a separate Telegram chat or WhatsApp:
1. `ncl groups create --name "Google One Cleanup" --folder google-one-cleanup`
2. `ncl groups config add-package --id <new-id> --apt python3` (+ pip)
3. Copy `audit.py` to `groups/google-one-cleanup/`
4. Write `groups/google-one-cleanup/CLAUDE.local.md` with the same audit behavior spec
5. `ncl wirings create --agent-group-id <new-id> --messaging-group-id <new-chat-id>`
6. Remove the audit section from `groups/dm-with-bigben/CLAUDE.local.md`

Nothing in `audit.py` changes — only where it lives and who runs it.
