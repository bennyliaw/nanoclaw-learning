# Plan: Use Case 1.2 Phase A — Google One Storage Audit (Nano-orchestrated)
_v2 — revised after Step 0 investigation; Photos API confirmed blocked_

## Context

Use case 1.2 Phase A: a chat-triggered audit that scans Google storage, identifies cleanup
candidates, and reports via Telegram. No deletions — read-only audit.

**Architecture:** Nano (dm-with-bigben) orchestrates. `audit.py` lives in Nano's group
folder. No new agent group or wiring needed. When you later want a separate Telegram chat
or WhatsApp routing, the migration is: create `google-one-cleanup` group, move `audit.py`
there, add a wiring — script unchanged.

**What this teaches:**
- OneCLI credential proxy for Google OAuth (proxy injects tokens; no auth code needed)
- Python3 in a NanoClaw container (requires apt packages + container rebuild)
- Agent-driven multi-step operation: run script → process output → send report
- Audit-before-act pattern
- Knowing when to pivot: recognising blocked API paths and adapting

## Step 0: Prerequisites — COMPLETED

### 0a. OneCLI provider support ✅
Both `google-photos` and `google-drive` are available providers in OneCLI.

### 0b. Connect services ✅
- Google Drive: connected (OAuth, `drive.readonly` + `drive.file` scopes)
- Google Photos: connected (OAuth, `photoslibrary` scope)
- Required: create Google Cloud project with own OAuth 2.0 credentials (Client ID +
  Secret) — self-hosted OneCLI does not bundle credentials. Redirect URIs:
  - `http://localhost:10254/api/apps/google-photos/callback`
  - `http://localhost:10254/api/apps/google-drive/callback`

### 0c. Photos API probe ✅ (confirmed blocked)
**Finding: Google Photos Library API is restricted for unverified apps (March 2025).**
- `mediaItems.list`, `mediaItems.search`, `albums.list` → all return 403
- Error: "Request had insufficient authentication scopes" (misleading — token IS correct,
  Google blocks bulk library access for unverified apps)
- OneCLI proxy IS injecting the Photos token correctly (Google error, not OneCLI error)
- See `docs/troubleshooting/google-photos-api-restriction-2025.md` for full details

**Drive API: working.** Storage breakdown confirmed:
- Total usage: ~205 GB (OVER QUOTA on 200 GB plan)
- Usage in Drive: ~570 MB (almost nothing)
- Remainder (~204 GB): Google Photos + Gmail

**What Phase A will deliver:**
- Storage breakdown (Drive + quota numbers)
- Gmail large attachments report (top offenders by size)
- Drive large files report

**What requires manual action (Photos):**
- Go to photos.google.com → settings → Manage Storage → Free up space
- Google's own tool finds duplicates, blurry photos, screenshots

## Step 3: Write `groups/dm-with-bigben/audit.py`

Script structure:

**Section 1 — Storage totals** (Drive `about` endpoint — already verified working):
```
GET https://www.googleapis.com/drive/v3/about?fields=storageQuota,user
```
Returns: `limit`, `usage`, `usageInDrive`, `usageInGmail`.

**Section 2 — Gmail large attachments**:
```
GET https://gmail.googleapis.com/gmail/v1/users/me/messages?q=has:attachment+larger:5mb&maxResults=20
```
For each message: get headers (subject, from, date, size).
Sort by size, report top 20.

**Section 3 — Drive large files**:
```
GET https://www.googleapis.com/drive/v3/files?orderBy=quotaBytesUsed+desc&pageSize=20&fields=files(name,size,mimeType,modifiedTime,webViewLink)
```

**Output:** Write `/workspace/agent/audit_results.json`. Print compact JSON summary to
stdout for agent to read.

**Auth:** No auth code needed — `requests` honours the `HTTPS_PROXY` env var injected
by OneCLI gateway. If a call returns 401/403, print the error JSON and exit 1.

**Bootstrap:**
```python
for pkg in ["requests"]:
    try: __import__(pkg)
    except ImportError:
        subprocess.check_call([sys.executable, "-m", "pip", "install", "-q", pkg], ...)
```

## Key Findings from Step 0 (learning log)

- **Self-hosted OneCLI requires own Google Cloud OAuth credentials** — not bundled
- **Google Cloud OAuth consent screen must be External + test user added** — even for personal use
- **Redirect URIs are app-specific** — `google-photos/callback` ≠ `google-drive/callback`
- **OneCLI routes by hostname** — `www.googleapis.com` (Drive/Gmail) works; routing for
  `photoslibrary.googleapis.com` injects the token correctly but Google blocks the API
- **Wiring ≠ Destination** — adding a new channel requires both `wirings create` AND
  `destinations add`; see `docs/troubleshooting/nanoclaw-wiring-vs-destination.md`
- **Container restart required after CLAUDE.local.md changes** — `docker kill <container>`
